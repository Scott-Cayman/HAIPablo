# 异步生成改造说明

## 背景

当前项目的图片生成流程是同步请求：

1. 前端直接调用 `/api/image/generate`
2. 服务端在一个请求里完成：
   - 预扣潮能力
   - 创建历史记录
   - 下载/整理参考图
   - 调用上游图片供应商
   - 保存结果图
   - 更新历史状态
3. 前端一直等待这个接口返回

这种方式实现简单，但在供应商响应时间较长时，会产生以下问题：

- 浏览器容易出现 `Failed to fetch`
- 反向代理、CDN、负载均衡层可能先于应用超时
- 批量任务会放大等待问题
- 前端刷新或关闭页面后，任务状态难以追踪
- 运维期或供应商抖动时，错误体验不稳定

## 现状问题分析

### 1. 同步链路过长

`/api/image/generate` 当前承担了完整任务生命周期，导致单次请求持续时间过长。

### 2. 前端状态与后端任务耦合

前端把“请求是否返回”当作“任务是否完成”的依据，这对慢任务并不适合。

### 3. 批量任务本质上还是多次同步请求

虽然前端做了批量调度，但每个子任务仍然是同步等待上游完成，因此本质问题没有消失。

### 4. 超时只能缓解，不能根治

把超时从 55 秒调到 150 秒，只是扩大等待窗口。如果供应商更慢，问题仍会出现。

## 目标方案

把图片生成改为“异步任务模式”：

1. 前端发起生成任务
2. 后端立即返回 `jobId`
3. 后端继续处理生成流程
4. 前端轮询任务状态
5. 任务完成后前端读取最终结果图

## 推荐任务状态

建议统一任务状态枚举：

- `queued`：已进入队列，尚未开始
- `processing`：正在处理
- `success`：成功完成
- `failed`：处理失败
- `cancelled`：已取消，可选

## 数据层建议

如果项目后续会长期维护，建议增加独立任务表，而不是只依赖 `GenerationHistory`。

### 推荐新增字段或表

可新增 `GenerationJob`，至少包含：

- `id`
- `userId`
- `templateId`
- `historyId`
- `status`
- `requestJson`
- `resultJson`
- `errorMessage`
- `providerId`
- `attemptCount`
- `startedAt`
- `finishedAt`
- `createdAt`
- `updatedAt`

如果你暂时不想建新表，也可以先在 `GenerationHistory` 上扩展：

- `status`
- `configJson`
- `errorMessage`
- `taskType`
- `providerId`
- `startedAt`
- `finishedAt`

但长期看，任务表和历史表分离会更清晰。

## 接口改造建议

### 第一阶段：提交任务

新增接口：

- `POST /api/image/jobs`

职责：

- 校验用户与参数
- 预扣潮能力
- 创建历史记录
- 创建任务记录
- 返回 `jobId`

返回示例：

```json
{
  "success": true,
  "jobId": "job_xxx",
  "historyId": "his_xxx",
  "status": "queued"
}
```

### 第二阶段：查询任务状态

新增接口：

- `GET /api/image/jobs/[id]`

职责：

- 返回任务状态
- 返回失败原因
- 返回结果图 URL

返回示例：

```json
{
  "success": true,
  "jobId": "job_xxx",
  "status": "processing",
  "progressText": "正在调用图片供应商"
}
```

### 第三阶段：执行器

生成逻辑从页面请求链路中拆出，放到任务执行器中。

执行器职责：

- 读取任务
- 调用现有图片生成逻辑
- 更新任务状态
- 更新 `GenerationHistory`
- 成功时写入结果图
- 失败时退还潮能力

## 执行器实现方案

### 方案 A：应用内轮询型执行器

适合现阶段快速落地。

方式：

- 提交任务后，启动一个后台异步处理流程
- 或由轮询器定时扫描 `queued` 任务再处理

优点：

- 改动相对可控
- 不依赖额外基础设施

缺点：

- 如果进程重启，需要做好未完成任务恢复

### 方案 B：独立 Worker

更适合长期生产使用。

方式：

- Web 应用只负责入队
- 单独 Worker 负责消费任务

优点：

- 最稳定
- 便于限流、重试、并发控制

缺点：

- 需要额外部署与运维

## 前端改造建议

目标页面：

- `app/generate/page.tsx`

改造方向：

1. 点击生成后，调用“提交任务接口”
2. 记录返回的 `jobId`
3. 进入“任务处理中”UI 状态
4. 每 2 到 5 秒轮询一次任务状态
5. 成功后展示结果图
6. 失败后展示明确错误信息

### 前端状态建议

- `idle`
- `submitting`
- `queued`
- `processing`
- `success`
- `failed`

### 用户体验建议

- 页面上显示“任务已提交，正在生成”
- 刷新页面后，如果有未完成任务，可以继续恢复查询
- 历史记录中的 `processing` 项应支持自动刷新

## 历史记录改造建议

目标文件：

- `app/api/history/route.ts`
- `app/generate/page.tsx`
- 历史列表相关页面

建议补充：

- `processing` 状态展示
- `failed` 失败原因展示
- 处理中任务自动刷新按钮
- 最近任务优先置顶

## 批量生成建议

批量生成后续建议不要继续由前端调度多个同步请求，而是：

1. 一次创建“批量任务”
2. 后端拆分为多个子任务
3. 前端轮询批量总进度

建议状态：

- `total`
- `completed`
- `successCount`
- `failedCount`
- `items`

## 实施步骤建议

建议分 4 个阶段完成：

### 阶段 1：任务模型落库

- 新增任务表或扩展现有表
- 补充状态字段
- 补充失败原因字段
- 补充开始/完成时间字段

### 阶段 2：新建任务接口

- 新增提交任务接口
- 新增任务详情接口
- 保留原同步接口作为兼容入口

### 阶段 3：前端切换

- `generate` 页面切到任务模式
- 历史记录增加处理中展示
- 批量页适配任务状态

### 阶段 4：移除旧同步模式

- 下线旧 `/api/image/generate` 同步直返模式
- 统一走任务队列
- 清理兼容逻辑

## 风险点

### 1. 潮能力扣费一致性

要保证：

- 任务创建成功才扣费
- 任务失败时准确退还
- 重试时不要重复扣费

### 2. 并发与重复提交

要防止：

- 用户连点多次造成重复任务
- 页面刷新后重复提交
- 批量模式重复触发

### 3. 任务恢复

如果服务重启，要能识别：

- `queued` 但未开始的任务
- `processing` 但实际中断的任务

### 4. 历史状态同步

任务表和历史表如果并存，需要保证状态一致。

## 最小可行改造路径

如果你想先做一个可上线版本，建议按下面最小路径推进：

1. 新增 `GenerationJob`
2. 新增 `POST /api/image/jobs`
3. 新增 `GET /api/image/jobs/[id]`
4. 前端生成页改成提交任务 + 轮询
5. 历史记录补 `processing`

这一步完成后，同步超时问题基本就从产品层面被解决。

## 当前结论

现阶段的同步生成方式已经能跑，但不适合上游响应时间不稳定或较慢的生产环境。

维护模式已经适合用来支撑上线运维，而异步生成则是下一步提升稳定性的核心工程项。

建议你后续优先级：

1. 先使用维护模式保障上线维护体验
2. 再逐步推进异步生成任务化改造
